# Multi-Agent Orchestration Framework

A multi-agent orchestration framework built on the Anthropic API. You describe a goal in natural language to a Claude Code session (the **parent**); the parent breaks it into tasks and dispatches them to **pod workers** (separate processes with their own API keys). Every task passes through the user twice — once before the pod runs (**before gate**) and once after (**after gate**) — so nothing moves without your explicit approval.

The full design is in [`METHODOLOGY_SIMPLIFIED.md`](./METHODOLOGY_SIMPLIFIED.md). This README is just how to run it.

## Three terminals

The framework is a backend + at least one pod + a parent (your Claude Code session). Open three terminals.

### Terminal 1 — bootstrap and start the backend

```bash
# Bootstrap a fresh framework-state directory pointing at your target repo
.venv/bin/python -m framework --state-dir ./fw run start \
  --goal "Implement the UCI handshake in engine.py" \
  --target-repo /path/to/your/local/clone \
  --overwrite

# Start the FastAPI backend on :8765
.venv/bin/python -m framework --state-dir ./fw backend
```

The bootstrap creates `./fw/` with `framework.db`, `config.yaml`, `CLAUDE.md` (the parent's role config), `agents/methodology.md`, `agents/development.md`, `agents/testing.md`, a starter `rolling_summary.md`, `progress.md`, and `run.yaml`.

### Terminal 2 — start a pod

```bash
export ANTHROPIC_API_KEY_POD_A=sk-ant-...
.venv/bin/python -m framework --state-dir ./fw start-pod pod_a
```

The pod registers itself, polls every 2 seconds, claims one approved task at a time, calls Claude with the agent's role config from `agents/<role>.md`, and submits the result back through the after gate.

### Terminal 3 — your Claude Code session (the parent)

```bash
cd ./fw      # so Claude Code picks up CLAUDE.md
claude
```

Now talk to it conversationally:

> "Run a methodology pass on the UCI goal."

The parent will translate that into framework tool calls (`framework subagent invoke methodology …`), surface the proposed plan, walk you through approving each task at both gates, and update `rolling_summary.md` after each after-gate approval.

## What the parent does for you

The parent's `CLAUDE.md` (auto-generated by `run start`) tells it the rules. Roughly:

1. Run `framework state` at the start of every turn to see pod status, queue counts, recent events, pending gates, and today's spend vs. cap.
2. At the **before gate**, surface the full TaskSpec to you. Apply any edits via `framework plan edit`. Approve with `framework gate before approve <task_id>`.
3. At the **after gate**, surface the full artifact body and the per-task stats block (model, tokens in/out, cost, duration). Approve with `framework gate after approve <task_id>` or reject with `--reason "..."` to send the task back to the before gate.
4. After every after-gate approval, update `rolling_summary.md` (capped at ~2000 tokens). The summary is the only running-state context that gets passed to pods — so keep it good.
5. Every framework tool call auto-logs to `logs/parent_actions.jsonl`. The parent doesn't log manually.

You can do all of this without Claude Code if you want — see "Direct CLI usage" below.

## Direct CLI usage

Every framework tool is a `python -m framework …` subcommand. Useful for one-off inspection or scripting.

### Inspection (read-only)

```bash
# Snapshot of pod status, queue counts, pending gates, today's spend
.venv/bin/python -m framework --state-dir ./fw state

# Read-only SQL against framework.db (SELECT/WITH/PRAGMA only)
.venv/bin/python -m framework --state-dir ./fw db query \
  "SELECT task_id, status, agent_role FROM tasks ORDER BY created_at DESC"

# List active tasks (archived ones hidden by default)
.venv/bin/python -m framework --state-dir ./fw plan show
.venv/bin/python -m framework --state-dir ./fw plan show --include-archived
.venv/bin/python -m framework --state-dir ./fw plan show --status before_gate

# Artifacts
.venv/bin/python -m framework --state-dir ./fw artifact list --task t_xxxxx
.venv/bin/python -m framework --state-dir ./fw artifact get a_xxxxx
```

### Planning + gates

```bash
# Have the methodology agent draft a plan from a goal
cat > /tmp/task.yaml <<'EOF'
goal: "Implement UCI handshake in engine.py"
target_repo: /path/to/your/repo
EOF
ANTHROPIC_API_KEY=sk-ant-... \
  .venv/bin/python -m framework --state-dir ./fw subagent invoke methodology /tmp/task.yaml
# → writes ./fw/plan/proposed_<ts>.yaml

# Create the tasks (lands them all in before_gate)
.venv/bin/python -m framework --state-dir ./fw plan create ./fw/plan/proposed_<ts>.yaml

# Edit a TaskSpec at the before gate before approving
.venv/bin/python -m framework --state-dir ./fw plan edit t_xxxxx priority 9
.venv/bin/python -m framework --state-dir ./fw plan edit t_xxxxx recommended_model claude-opus-4-7

# Gate transitions
.venv/bin/python -m framework --state-dir ./fw gate before approve t_xxxxx
.venv/bin/python -m framework --state-dir ./fw gate before reject t_xxxxx --reason "too vague, will respec"
.venv/bin/python -m framework --state-dir ./fw gate after approve t_xxxxx
.venv/bin/python -m framework --state-dir ./fw gate after reject t_xxxxx --reason "wrong file edited"
```

### Maintenance

```bash
# Replace the rolling summary (do this after every after-gate approval)
.venv/bin/python -m framework --state-dir ./fw summary update /tmp/new_summary.md

# Long run getting cluttered? Archive done/rejected tasks (audit trail preserved)
.venv/bin/python -m framework --state-dir ./fw session reset

# Pod was killed mid-task and a task is stuck in 'claimed'?
.venv/bin/python -m framework --state-dir ./fw task requeue t_xxxxx
```

## Configuration

`./fw/config.yaml` (auto-created by `run start`):

```yaml
budget:
  daily_cap_usd: 50.00     # claim_next_task refuses once today's spend hits this
models:
  haiku: claude-haiku-4-5-20251001
  sonnet: claude-sonnet-4-6
  opus: claude-opus-4-7
  methodology_default: claude-opus-4-7
rolling_summary:
  max_tokens: 2000
retries:
  per_call: 3              # SDK-handled exponential backoff
pricing:                   # USD per million tokens
  claude-opus-4-7:           {input: 5.00, output: 25.00}
  claude-sonnet-4-6:         {input: 3.00, output: 15.00}
  claude-haiku-4-5-20251001: {input: 1.00, output: 5.00}
pod:
  max_tokens: 4096
  idle_sleep_seconds: 2.0
```

The backend re-reads `config.yaml` on every request, so you can edit and reload without restarting.

## Files the framework writes

```
fw/
├── framework.db             # SQLite source of truth (Invariant #6)
├── events.jsonl             # append-only event stream
├── CLAUDE.md                # parent role config (read by Claude Code)
├── config.yaml              # see above
├── run.yaml                 # goal, target_repo, branch_name, run_id
├── rolling_summary.md       # the only running-state context passed to pods
├── progress.md              # global change log
├── agents/
│   ├── methodology.md       # role configs (read fresh per task)
│   ├── development.md
│   └── testing.md
├── plan/
│   └── proposed_*.yaml      # methodology agent output
└── logs/
    ├── parent_actions.jsonl # every framework tool call (auto-logged)
    ├── budget_ledger.jsonl  # one line per pod API call
    ├── methodology_agent.md # per-agent change logs (Phase 5)
    ├── development_agent.md
    └── testing_agent.md
```

## Architectural invariants

The 14 invariants in [Section 21 of the methodology](./METHODOLOGY_SIMPLIFIED.md#21-architectural-invariants) are the load-bearing constraints. Most importantly:

- **Every task passes through both gates.** No exceptions.
- **Pods accumulate no context across tasks.** Every call is a fresh `messages.create`.
- **The parent reads framework state from disk every turn.** Don't trust conversational memory.
- **All claim logic goes through `claim_next_task`.** Don't reimplement.
- **Backend (SQLite) is the source of truth.** Files are mirrors.
- **Edits to TaskSpecs go through `framework plan edit`.** Never edit YAML or SQLite directly.

If you find yourself wanting to violate one, stop and reconsider.
